# Contributing to Spark

Thank you for contributing to Spark — The Global Stage. We welcome all contributions that align with our vision and principles.

## Development Workflow

1. **Fork and clone** the repository.
2. **Create a feature branch** from `main`: `git checkout -b feat/my-change`.
3. **Make your changes** following the code style and conventions of the project.
4. **Write tests** for all new functionality and ensure existing tests pass.
5. **Run linting and type-checking** before committing.
6. **Commit using conventional commits**: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`.
7. **Open a pull request** against `main` with a clear title and description.

## Pull Request Standards

- Every PR must link to an issue.
- PRs must pass CI (lint, test, build, type-check).
- All new code must include tests (unit, integration, or E2E as appropriate).
- Documentation must be updated for any exposed API or behavioral change.
- A PR requires at least one approval from a code owner or maintainer.

## Code Review

Reviewers evaluate correctness, security, performance, test coverage, and adherence to Spark's engineering principles. Every review comment is a learning opportunity — respond constructively and promptly.

## Branch Naming

- `feat/*` — new features
- `fix/*` — bug fixes
- `chore/*` — maintenance
- `docs/*` — documentation
- `refactor/*` — code restructuring
