# Governance Model

Spark is governed as an **open-core project** with a transparent, community-driven decision-making process.

## Roles

### Contributors
Anyone who submits a pull request, reports a bug, or participates in discussions. No formal onboarding required.

### Maintainers
Trusted contributors with write access to the repository. Responsibilities include reviewing PRs, triaging issues, and ensuring code quality. Maintainers are nominated by existing maintainers and approved by consensus.

### Core Team
Senior maintainers who oversee architecture decisions, release management, and the project roadmap. The core team votes on strategic direction and major breaking changes.

### Steering Committee
Focused on governance, community health, funding, and legal matters. The committee resolves disputes that escalates beyond the core team.

## Decision-Making

| Decision Type | Process |
|--------------|---------|
| Bug fixes, minor features | Maintainer approval |
| New packages, breaking changes | Core team consensus |
| Roadmap, license changes | Steering committee vote |
| Code of conduct violations | Steering committee |

## Voting

- Standard decisions require a simple majority of maintainers.
- Strategic decisions require a two-thirds supermajority of the core team.
- Each maintainer has one vote. The steering committee chair breaks ties.

## Communication

All governance discussions happen publicly in GitHub Discussions unless they involve sensitive security or conduct matters. Meeting notes and decision logs are published in `docs/governance/`.
