# Vector Database Strategy

Spark uses vector databases for similarity search across embeddings generated by AI models — powering recommendations, content discovery, moderation, and search features.

## Primary Store — Qdrant

Qdrant serves as Spark's primary vector database due to its performance, filtering capabilities, and Rust-based efficiency.

### Use Cases
- **Content Similarity Search**: Find visually or semantically similar videos using CLIP embeddings. Powers "more like this" recommendations and duplicate content detection.
- **User Embedding Matching**: Match user interest vectors against content embeddings for candidate generation in the recommendation pipeline.
- **Moderation Hash Database**: Near-duplicate detection for previously moderated content using perceptual hashes stored as vectors.
- **Semantic Search**: Natural language search over content using text embeddings (E5-large-v2). Supports hybrid search with keyword BM25 fallback.

### Configuration

| Parameter | Value | Rationale |
|---|---|---|
| Embedding Dimension | 768 | CLIP ViT-L/14 output dimension |
| Index Type | HNSW | Best balance of recall, latency, and memory |
| Distance Metric | Cosine | Normalized embeddings, bounded [−1,1] |
| Segment Size | 100,000 | Optimal for HNSW graph construction |
| Replication | 3x | Fault tolerance and read throughput |

## Secondary Store — FAISS

FAISS is used for high-throughput batch similarity computations (millions of comparisons) including training data generation via hard negative mining, offline recommendation candidate pool refresh, and large-scale deduplication during ingestion.

## Hybrid Search Strategy

For search queries, Spark combines vector similarity with metadata filtering using Qdrant's payload filter mechanism. Filters include content type, language, upload date, creator tier, and safety labels. This provides accurate and policy-compliant results without post-filtering inefficiencies.
